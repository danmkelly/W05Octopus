The Pipeline

The Control solution received only the minimal prompt — a ten-line brief with no supporting research, no customer data summaries, and no strategic guidance. The agents had to derive everything from the raw CSV alone.

Yellow Researcher ─▶ Red-Orange Designer ─▶ Blue Maker ∥ Green Marketer ─▶ Purple Manager (Scout mode) (Design spec) (Build page + Write copy) (Integrate & ship)

Agent-by-Agent Breakdown

Stage 1 — Intelligence

🟡 Yellow Researcher & Analyst

Diagnosed the churn by running PowerShell GROUP BY queries on the 45,349-row CSV. Identified 655 one-and-done members (42.1%) with a 75.3% cancellation rate. Found that first-class no-show is the #1 churn predictor (69.4% cancel rate), and that the 1st-to-2nd booking gap is a measurable early-warning signal (active median 5 days vs cancelled median 11 days). Named the at-risk segment: 158 active one-and-done members representing ~€7,000–11,000/month in recurring revenue.

Artifacts: control-churn-analysis.md (379 lines)
Stage 2 — Design

🟠 Red-Orange Designer

Designed a single-page progressive-disclosure booking experience targeting three churn moments: one-and-done, first-class no-show, and 7+ day inactive. Produced an 836-line design spec with ASCII wireframes, component specs, interaction states, and a traceability matrix linking every design decision to a specific research finding. Key design decisions: progressive disclosure on one page (no multi-page wizard), 7-day date window only, "Recommended" badges on Pilates and HIIT, and "Save My Spot" CTA language.

Artifacts: control-retention-page-spec.md (836 lines)
Stage 3a — Build

🔵 Blue Maker

Built a 1,678-line, 53.8 KB single-page HTML booking experience. Implements progressive disclosure (Hero → Class Picker → Date/Time → Confirmation → Success), three entry-context hero variants via URL params, 7 class types with embedded schedule JSON, pre-filled identifiers from URL params, animated SVG checkmark success state, client-side ICS calendar generation, and responsive design at three breakpoints. No external dependencies beyond Google Fonts.

Artifacts: control/index.html (1,678 lines) · maker-done.md
Stage 3b — Copy

🟢 Green Marketer

Wrote 321 lines of sales copy covering three hero variants (at-risk: "We saved your spot", no-show: "Fresh start. New class.", inactive: "Your next class is waiting"), all 7 class type cards with intensity indicators and "why try this" taglines, section headers and form microcopy, five "Good to Know" post-booking tips, meta tags, and a three-line social proof strip. Built a full research-to-copy traceability matrix.

Artifacts: marketer-copy.md (321 lines)
Stage 4 — Orchestration

🟣 Purple Manager

Integrated the Marketer's copy into the Maker's page: added social proof strip, class card descriptions with intensity dots, updated all five tips to match the copywriter's exact language, added meta tags and OG tags, updated form placeholders and error messages, added success subheading, and refined the hero copy. Committed and deployed to GitHub Pages.

Issues & Concerns

Spec Deviation: 7 Class Types vs 4

The design spec's wireframes showed 4 class types (Pilates, HIIT, Yoga, Spin) but the Maker was instructed to include all 7 from the data. The grid layout handled it, but this is a spec-vs-reality mismatch — the designer worked from a simplified model while the builder had the full data.

No Real Backend

The form submission is simulated with a 1.5s delay. A TODO comment marks where the real API call would go. In production, this would need a Formspree or equivalent backend for the booking to actually reach the gym.

Static Schedule

The class schedule is hardcoded JSON in the page. For real use, a dynamic schedule API would be needed to keep slots accurate week-to-week.

No Cold Verify Gate

Unlike the Real Brief pipeline, the Control pipeline had no independent audit step. The Manager reviewed the outputs but there was no blind verifier checking the deliverables against frozen criteria. This means undetected errors (hallucinated stats, broken handoffs, off-brand lines) could have slipped through uncaught.

Advantage: The minimal prompt forced the agents to derive facts from the raw data — no risk of hallucinating from assumed summary statistics. Every number came from the CSV.
Disadvantage: Without the rich harness (brand voice, member verbatims, three.js spec, tone discipline), the Control page is a functionally solid but emotionally generic booking form. It books a class, but it doesn't tell a story.
